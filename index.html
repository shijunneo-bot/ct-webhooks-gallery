Ground truth

What a webhook is, in four cards

Skim these and you have the whole model: what it is, the flavors, how reliable it is, and what it can power.

What it is

The outbound channel. CleverTap talks to the client's own stack, server to server, when an event or campaign fires. Not a user-facing surface.

Grounded: developer.clevertap.com, API overview.

The flavors

Four: a webhook campaign (an HTTP call as the delivery channel), a Journey webhook action node, a Post Action Webhook fired after a delivered campaign, and the inverse External Trigger where an outside system fires a CleverTap campaign.

Grounded: docs.clevertap.com, plus clevertap-api skill (External Trigger).

The reliability model

It expects a 200 OK. The published figure is a 5-second timeout with a retry limit of 2. Internal CS guidance has cited 3 retries at about 3-second intervals. The numbers conflict, so design for very few retries and no dead-letter, and confirm the current figure.

Grounded: docs.clevertap.com webhooks FAQ (5s, 2 retries), with an internal-guidance variant. Verify with PM.

What it can power

Ops and chat alerts, CRM and CDP sync, the brand's own inbox, coupon and loyalty updates, warehouse feeds, real-time personalization, and fraud or risk scoring.

Grounded patterns across CleverTap docs and partner integrations.
The payload, up close

One POST, templated to the user

This is the shape CleverTap sends: an HTTP POST with a JSON body you build from Liquid macros. Green is a field you template. Amber is a field the platform sets for you.

POST https://hooks.examplebrand.example/ct/events
Content-Type: application/json
X-CT-Signature: <hmac-sha256 of raw body>
Authorization: Bearer <token>
{ "event": "Cart Abandoned", "ts": "{{ System.Timestamp }}", "source": "clevertap", "profile": { "objectId": "{{ Profile.objectId }}", "identity": "{{ Profile.Identity }}", "name": "{{ Profile.Name | default: 'there' }}", "email": "{{ Profile.Email | default: 'n.a' }}" }, "event_props": { "cart_value": "{{ Event.cart_value | default: 0 }}", "currency": "SGD", "items": "{{ Event.item_count | default: 0 }}" } }
Templated (Profile or Event macro) System-set (objectId, timestamp) Keys you define
Verify with PM. Macro namespaces (Profile, Event, System) and the exact signing header can vary by account and version. Confirm the current shape before you wire a client.

Anatomy of the config

What you set, and what the receiver owns

Gold is set in the CleverTap dashboard. Green lives on the receiver side, in the client's own code.

Dashboard-set Receiver-side

Destination URL

The HTTPS endpoint the payload flies to. Add query params as key-value pairs if needed.

HTTP method

POST, GET, PUT, or DELETE. POST is the workhorse for JSON payloads.

Headers

Content-Type, plus an Authorization or signature header the receiver checks.

Authentication

None, Basic, OAuth 2.0 (client-credentials or password grant), or a Bearer token via custom header.

Content type

Plain text or JSON. JSON for anything talking to a real API.

Payload body

Templated with Liquid macros: Profile fields, Event properties, and system fields like objectId and timestamp.

Trigger

Campaign target, event, Journey node, or schedule. Live behavior for real-time, past behavior for periodic.

Signature verify

The receiver recomputes an HMAC over the raw body and compares in constant time before trusting the payload.

Retry and timeout

The receiver must ACK with 200 fast, then work async. Idempotency on event plus objectId protects against a retry landing twice.

Dead-letter

A queue on the client side catches anything the endpoint could not process, so nothing is lost silently.

Reliability and security

The trust section

A webhook is only as good as its weakest promise. Here is the honest model: retries, the data-loss cliff, idempotency, and how a receiver proves the payload is real.

Expect a 2xx

CleverTap treats a 200 OK as success. Anything else, or a timeout, counts as a failure and starts the retry clock.

Retries are limited

The published FAQ states a 5-second timeout with a retry limit of 2. Internal CS guidance has cited 3 retries at about 3-second intervals. The figures conflict, so treat the exact number as Verify-with-PM and design as if you get very few.

There is a data-loss cliff

If the endpoint does not ACK with a 2xx across the retry window, the event is dropped. There is no dead-letter on CleverTap's side, so build one on yours.

It is fire-and-forget

CleverTap does not read the response body. To act on what the receiver returns, close the loop with the Events or Profile API, or use the inverse External Trigger.

Order is not guaranteed

Do not assume events arrive in sequence. Carry a timestamp in the payload and let the receiver sort.

Make the receiver idempotent

A retry can deliver the same event twice. Dedupe on a stable key (event name plus objectId plus timestamp) so a double delivery is harmless.

Verify authenticity

Sign the payload (HMAC over the raw body) or protect the endpoint with a secret header, and allowlist CleverTap's published IP ranges for inbound traffic.

Watch the stats

CleverTap reports webhooks streamed, dispatch errors, and invalid-URL errors. That confirms it fired, not that the downstream delivered. Instrument the receiver for the rest.

Verify with PM. The exact retry count, timeout, and signing scheme move between docs and versions. Design as if you get very few retries and no dead-letter, then confirm the current numbers.
Watch-outs

Six things to say out loud before you build

None of these are dealbreakers. They are the honest edges, the ones that save a client from a fragile setup if you name them early.

DND does not work inside Journeys

Do Not Disturb for webhooks is supported in Campaigns only, not in Journeys. If you need quiet hours inside a journey, enforce them in your own middleware.

Fire-and-forget, no response parsing

CleverTap does not read what your endpoint returns. To act on a response, write it back with the Events or Profile API, or use the External Trigger.

No dead-letter on CleverTap's side

If your endpoint is down past the retry window, the event is gone. Put a queue in front of anything that matters so nothing is lost silently.

Retry and timeout figures conflict

The published FAQ says 5-second timeout and a retry limit of 2. Internal CS guidance has cited 3 retries at about 3 seconds. Verify with PM and design for the pessimistic case.

Campaign targeting cap is 100 or more

Webhook and External Trigger campaigns need a targeting cap of at least 100, regardless of the real segment size, or the campaign throws an error.

Order is not guaranteed

Do not assume events land in sequence. Carry the timestamp in the payload and let the receiver sort and dedupe.

The inverse: External Trigger

When the outside world fires the campaign

The inverse of a webhook. Instead of CleverTap pushing out, an outside system calls CleverTap to fire a campaign you already built, and hands it fresh data to drop into the message before it sends.

Public Beta. Public Beta. Ask your CSM to enable it. You publish the campaign first to get a Campaign ID, then your servers call the trigger API with that ID.
Channels it can fire
PushEmailWhatsAppWebhook
How it runs
  1. Build the campaign in CleverTap and pick the channel (Push, Email, WhatsApp, or Webhook).
  2. Set the qualification to External Trigger. It targets a user ID, and you can still filter by past behavior or user properties.
  3. Write the message with ExternalTrigger Liquid tags, so the API payload fills the blanks at send time.
  4. Publish to get a Campaign ID, then fire the trigger API from your backend with that ID and the supplementary data.
Watch-outs
  • It is Public Beta, so availability and behavior can change. Verify with PM.
  • The targeting cap must be 100 or more, or the campaign errors.
  • You need the Campaign ID from publishing before the trigger API will work.

Choose the right tool

Webhook is one option, not the only one

Before you reach for a webhook, check it against the alternatives. Sometimes a connector, an export, or a journey action is the cleaner fit.

Build vs buy

Three lanes, one honest cost

Buy

Use a native connector where one exists

If CleverTap has a real connector for the platform (a CDP, a warehouse path, a messaging partner), start there. Least plumbing, most support. This is the default for anything on the supported list.

Hybrid

Webhook to your own service

Point a webhook at a small service you own, which then fans out to the real destination. This is the right starting point for most custom integrations: fast to stand up, and you control the contract.

Build

Own the full pipeline

For high volume or strict guarantees, build the event pipeline end to end (gateway, queue, warehouse). Honest cost: you now own retries, backoff, dead-letters, and monitoring that CleverTap will not do for you.

FAQ

Twenty straight answers

Setup, payloads, retries, security, and limits. Where a fact depends on plan or version, it says Verify with PM.

Is a webhook a customer-facing channel?

No. Every other channel in the family talks to a user. A webhook talks to another system, server to server. It is how CleverTap reaches the brand's own stack.

How many kinds of webhook does CleverTap have?

Four in practice: a webhook campaign, a Journey webhook action node, a Post Action Webhook (fires after a delivered campaign), and the inverse External Trigger. Verify the current set with PM for your account.

What does the payload look like?

JSON you template with Liquid macros. You pull in Profile fields, Event properties, and system fields like objectId and the timestamp. You decide the shape to match the receiver's contract.

Which HTTP methods are supported?

POST, GET, PUT, and DELETE. POST carries JSON payloads and is the common choice.

What authentication can I use?

None, Basic, OAuth 2.0 (client-credentials or password grant), or a Bearer token through a custom header. Match the receiver's requirement.

How many times will CleverTap retry?

The published FAQ states a 5-second timeout with a retry limit of 2. Internal CS guidance has cited 3 retries at about 3-second intervals. The numbers conflict, so design as if you get very few and never depend on a long retry window. Verify with PM.

What happens if my endpoint is slow or down?

If it does not ACK with a 2xx inside the retry window (roughly 12 seconds total), the event is dropped. There is no dead-letter on CleverTap's side, so add a queue on yours.

Can CleverTap read my endpoint's response?

No. It is fire-and-forget. To act on a response, write the result back with the Events or Profile API, or use the External Trigger to fire a CleverTap campaign from outside.

How do I stop a retry from acting twice?

Make the receiver idempotent. Dedupe on a stable key such as event name plus objectId plus timestamp, so a repeated delivery is a no-op.

How should the receiver verify the payload is really from CleverTap?

Recompute an HMAC over the raw body and compare in constant time, or check a shared secret header, and allowlist CleverTap's published inbound IP ranges. Verify the exact signing scheme with PM.

Do webhooks respect Do Not Disturb?

DND applies to webhooks in Campaigns, not in Journeys. If you need quiet hours inside a journey, enforce them in your own middleware.

Can I reach LINE, KakaoTalk, Zalo, or Viber with a webhook?

Yes for LINE via a documented pattern, and the rest via a generic webhook to that platform's API or a provider. No native channel exists for them, so treat non-LINE ones as custom builds and scope the work. Verify with PM.

Is WhatsApp a webhook use case?

Usually not. CleverTap is a WhatsApp BSP, so WhatsApp is a native channel. Webhook-to-BSP survives mainly as a legacy or number-ownership case.

What is a Post Action Webhook?

A webhook that fires after a campaign is delivered, streaming message-level detail (which notification, which deeplink, delivery signals) to your endpoint. Useful for feeding a model or a warehouse.

What is the External Trigger?

The inverse. An outside system calls CleverTap to fire a pre-built campaign, injecting fresh data (an order id, a price) into the template before send.

Webhook or batch export for my warehouse?

Both. Webhooks for minutes-fresh, real-time triggers; S3 or GCP exports for comprehensive daily depth. Many teams run the two together.

Will a webhook slow my campaigns down?

Not if the receiver ACKs fast. Do the real work off the request path. A slow receiver is the usual cause of drops, not CleverTap.

Can one event trigger several systems?

Yes. Point the webhook at a small fan-out service you own, or place several webhook nodes in a Journey. Keep each receiver idempotent.

What does CleverTap show me about delivery?

Webhooks streamed, dispatch errors, and invalid-URL errors. That is proof it fired, not proof the downstream delivered. Instrument the receiver for end-to-end truth.

When is a webhook the wrong tool?

When a native connector already exists (use it), when you need bulk historical data (use exports), or when the action belongs inside a user journey (use a Journey action). Right tool, right job.
